# Sistema de Internacionalización (i18n)

Este proyecto utiliza `next-intl` para el manejo de múltiples idiomas tanto en la Landing Page como en el Dashboard.

## Cómo agregar un nuevo idioma

Para agregar un nuevo idioma (por ejemplo, Portugués `pt`), sigue estos pasos:

### 1. Configurar el routing
Edita el archivo `frontend/src/i18n/routing.ts` y agrega el nuevo locale a la lista de `locales`:

```typescript
export const routing = defineRouting({
  locales: ['es', 'en', 'ja', 'pt'], // Agregado 'pt'
  defaultLocale: 'es'
});
```

### 2. Crear el archivo de traducciones
Crea un nuevo archivo JSON en `frontend/src/locales/pt.json`. Puedes copiar el contenido de `es.json` y traducir las llaves.

> [!IMPORTANT]
> Asegúrate de que las llaves coincidan exactamente entre todos los archivos de idiomas para evitar errores de renderizado.

### 3. Actualizar el Dashboard (Formularios)
Para que los campos traducibles (como nombres de categorías o descripciones) muestren la pestaña del nuevo idioma en el Dashboard, edita `frontend/src/features/dashboard/components/FormGenerator.tsx`:

Busca la sección de los botones de idioma y agrega el nuevo:

```tsx
{['es', 'en', 'ja', 'pt'].map(lang => (
    <button ...>
        {lang === 'es' ? '🇪🇸' : lang === 'en' ? '🇺🇸' : lang === 'ja' ? '🇯🇵' : '🇧🇷 Portugués'}
    </button>
))}
```

### 4. Middleware (Opcional)
El middleware ya está configurado para detectar los locales definidos en `routing.ts`, por lo que no debería requerir cambios adicionales a menos que se necesiten reglas de redirección específicas.

## Traducciones dinámicas desde el Backend
Muchos campos en la base de datos (Categorías, FAQs, etc.) se guardan como objetos JSON de tipo `Record<string, string>`. 

Ejemplo de estructura en DB:
```json
{
  "es": "Electrónica",
  "en": "Electronics",
  "ja": "電子機器",
  "pt": "Eletrônicos"
}
```

El componente `getTranslation` en el frontend se encarga de seleccionar la llave correcta basada en el locale activo o caer en `es` como fallback.

# Otras Documentaciones

## Iconos Dinámicos
Usamos un sistema híbrido en `frontend/src/components/atoms/DynamicIcon.tsx`:
- **Custom SVGs**: Ubicados en `public/icons/svg/`. Se activan si el nombre coincide con la lista `ALL_CUSTOM_ICONS`.
- **Lucide Icons**: Fallback automático si el nombre coincide con cualquier icono de la librería `lucide-react`.

## Visualización Condicional
Las secciones de la Landing Page (`CategoryShowcase`, `FAQ`, `TestimonialWall`) están configuradas para retornar `null` si no hay datos en la API, manteniendo la página limpia sin secciones vacías.
